1. Introduction

1.1 Purpose & Product Scope

The product whose requirements are specified in this document is Database on Web. We propose a mobile-optimised version of well-known web-based relational database administration systems such as Adminer, phpMyAdmin or collaborative ones, such as DB Fiddle. The present document addresses only the end-user front-end workflow requirements.

1.2 Product Scope

The specified software product is intended as a robust, practical and mobile tool for managing relational databases, regardless of the used terminal (smartphone, tablet or desktop PC).

2. Overall description

2.1 Product perspective

The context in which the specified product is intended for operation is the world-wide web, which doesn't possess yet a comfortable enough database management system to be used on every terminal (especially smartphones).

2.2 Product functions

The application has the following features:

  • account-based access
  • account administration functions: reset password, change e-mail, delete account, etc.
  • mail notifications regarding future products and surveys
  • collaborative editing
  • basic CRUD operations on data
  • SQL command/script execution

2.3 User Classes and Characteristics

We expect the product's users to have a certain degree of expertise regarding the relational database model and the SQL language.

2.4 Operating Environment

The software will operate within any modern web browser, provided they are updated to their latest versions.

2.5 Design and Implementation Constraints

The software application's interface had to be written entirely from scratch using HTML5, CSS3 and vanilla JavaScript out of educational purposes.

2.6 User Documentation

The user's manual will be provided as the application continues to grow and will be shipped with the finished product.

2.7 Assumptions and Dependencies

The application is constrained by newer HTML and CSS standards and therefore expects the user to run it on a compatible browser. The user's manual will include minimum versions of popular browsers required to run the product properly.

3. External Interface Requirements

3.1 User Interfaces

3.1.1 Login page

The first page an unauthenticated user should see is the login page, where he can enter his credentials and proceed to the home page of the application. If he doesn't have any account, he should be able to register from the same page. Successful registration is followed by another login attempt.

3.1.2 Home page

  • The pages following the home page should contain a persistent navigation bar with buttons for returning to the home page, changing account settings, toggling between a light and a dark theme, collaborating with someone (generating an unique link for another registered users' access) and logging out.
  • On the left side of the home page a database list (on mobile devices, it should be hidden, except when floating over the page only after toggling its visibility with a button placed immediately above, on the navigation bar). From it, the user can select a database or find one by its name.
  • The rest of the page is filled with a floating command panel (bar) including (but not limited to) the following commands: open an SQL prompt, querying and searching through a table using the GUI, database import and export functionality, viewing all table views; the page should also contain tables from the selected database, having their fields described, each within its card. Each such card contains at the bottom buttons for dropping the table, altering the corresponding table and viewing its data.
  • An user should be able to create new tables on this page by entering their name and number of columns, and, at a later point in time, specify the definitions of its columns. He should also have the possibility of creating a new database on the same page.
  • In the page footer, a contact button should display on pressing it the authors' contact info.

3.1.3 SQL command prompt

The SQL prompt should be separated in two parts: one for introducing commands and another one for their computed output. They should be separated by a run button. The user should also have the option to display the results as a table (to be implemented at a later date).

4. System Features

4.1 Account-based access

Application's services should be available only after registration and each user must have his databases and associated information in a private sandbox.

4.2 Account administration functions

Each user must be able to change his (her) mail, password, name and e-mail subscription settings and delete their own accounts.

4.3 Collaborative editing

2 users must be able to manage a database in parallel via a randomly-generated, expiring link.

4.4 Basic CRUD operations on data

Each user must be able to Create, Read, Update or Delete (CRUD) any database, table, column or data within a row.

4.5 SQL command/script execution

Each user must be able to enter valid SQL commands and have them executed and the output displayed in the same page after selecting a database.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

  • the web application must have a responsive design (should render properly on any platform: smartphone, tablet, laptop or desktop PC)

  • the software product should score at least 90 points on PageSpeed Insights

5.2 Security requirements

  • each user should exercise precaution when committing changes to his (her) database as these are not undoable (they should be asked twice when pressing such a button)

  • each user must be aware that any potential data breach poses a real threat to any sensitive data stored in their database(s) and the authors assume no additional responsability when the user uploads such data
